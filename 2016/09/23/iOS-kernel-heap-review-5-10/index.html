iOS kernel heap review 5-10

Sep 23, 2016 | Hits

These article is originally from Stefan Esser’s iOS10 kernel Heap Revisited. I record it with my understandings and for future reference.

Kernel Zone Heap Allocator

Some features:

  • most used heap allocator in kernel
  • memory is divided into zones
  • zones group allocations of same type/size together
  • all allocations in one zone are same size
  • caller decides what zone is allocated from/free to
    • ptr = zalloc(zone)
    • zfree(zone, ptr)
  • size of allocated block depends on zone
  • no variable length allocations

What is Zone memory?

Kernel reserves a memory region called zone_map for zone allocator, the default size of this region is 1/4 of the physical memory, it’s reserved very early on during kernel start.
Pages from zone_map are assigned to zones, zones are grown by their allocation size. Kernel pages are 16kb on new 64 bit device with lots of memory and 4kb on all other devices. To waste less space zones use optimal allocation sizes >= 1 page.

Dynamic Length Allocations?

Zone allocator is not suited for dynamic length allocations. But there’re various wrappers to do this. e.g.:

  • kalloc(size)/kfree(ptr,size) - wrapper around zalloc()
  • MALLOC(size)/FREE(ptr) - wrapper around kalloc()

Kernel Heap Allocation Debugging

iOS has kernel boot arguments to enable heap allocation debugging.

  • -zc selects zone corruption logging mode
  • zlog= selects ONE zone to log
  • zrecs= controls how many log entries should be kept
    But need a kernel exploit to activate kernel boot arguments.[^1]

Kernel zone allocator corruption logging logs for all (de)allocations: if allocation or deallocation; pointer allocated/deallocated; kernel backtrace with up to 15 elements. Data is collected via btlog routines in kernel. Need to create custermized routine to extract data from kernel.

Zone Struct

Zone structs are stored in heap zone “zones”

Free Memory Blocks

  • free elements are kept in a single linked freelist per zone
  • zone structure has free_elements pointer to head of freelist
  • free elements have a pointer to next freeelement in beginning
  • there is no pointer(-chain) back to zone struct
    • not possible to know what zone an element (a page) belongs to
    • slow garbage collection

Allocation

allocation will return next element from freelist, when free list is empty, zone grows by allocation size. All elements in added pages are added to free list. Last element will be first in freelist.

Dynamic, kalloc()

kalloc() registers multiple zones, named kalloc., e.g. kalloc.128 for 128 allocations.
Allocations will be put into the next larger zone, e.g. 97 bytes allocations in kalloc.128.
Different iOS version, different bit(32/64) define different zones.
Caller needs to remember allocation size so that kfree(ptr,size) can put it back into the right zone.

Dynamic, MALLOC()

MALLOC() stores the size as meta data and stores the size in front of data. Internally uses kalloc()/kfree().

Attacks to iOS5 Zone Allocator

Heap-Feng-Shui, exploit specific allocation and deallocation primitives(e.g. opening/closing NDRV sockets); generic allocation deallocation primitives via OSUnserializeXML(controlling memory layout via many IOKit API functions using XML - and filling it with objects and object pointers)
Corruption Targets, zone allocator freelist next_ptr pointers(control ptr next allocation will return and overwrite with attacker controlled data); size fields of elements allocated via MALLOC() or a wrapper(tricks kfree() to put element into a zone that is for bigger elements - next allocation will
result in a larger bufferoverflow); application data(control objects/vtable pointers, object pointers, size fields, …)

Changes

iOS 6

  • Apple made some changes to OSUnserializeXML that had little impact on usability for heap feng shui.
  • location of zone_map is randomized due to KASLR
  • single linked freelist now protected by heap cookies/canaries
    • iOS creates two random cookies
      • zp_nopoison_cookie(lowest bit cleared)
      • zp_poisoned_cookie(lowest bit set)
    • last bytes of a free element will be overwritten with ptr^cookie
    • allocation code will detect illegal cookie values and panic()
    • protection against overflows and other ptr corruption
  • small free memory blocks are now poisoned
    • small blocks when freed get overwritten with Oxdeadbeef
    • indicated by use of zp_poisoned_cookie
    • on allocation 0xdeadbeef is verified - panic() if modified
Attack

Heap-Feng-Shui
Create vm_map_copy_t structures by sending mach messages with OOL data became heap-feng-shui method of choice.
Previously used methods less often seen but still possible.
Corruption Target
Target vm_map_copy_t structure overwrites, could be used for arbitrary info leak and kfree() confusion.

iOS 7

  • Poisoning larger blocks every n frees
  • Introduction of new heap pagelist feature(for easier GC)
  • zp_factor debugging feature(that already existed existed before) is now always activated
  • randomly set once at boot time to the value 15(25%), 16 (50%) or 17 (25%) (n?)
  • Controls after every how many frees in a zone a single block is poisoned(regardless of size)
  • Counter is zone specific in the zp_count field of the zone structure

  • Introduces new pagelist feature

    • activated via bit in zone struct (only subset of zones use new feature)

    • adds meta data at end of all pages inside a zone

      • meta data
        • backpointer to zone
        • free elements in this page(page local freelist)
        • forward and backward pointer to other pages
        • alloc_count - max number of elements in page
        • free_count - number of free elements in page

    • keeps all pages in one of four double linked lists (queue)

    • Allocator defines 4 queues for every zone
      • any_free_foreign - for the few zones that allow foreign elements
      • intermediate - for pages that are partially allocated at the moment
      • all_free - for pages that are completely free at the moment
      • all_used - for pages that are completely allocated at the moment
    • during allocation or free the allocator ensures that page is always on the right queue, allocation now traverses the page queues in the following order:
      • any_free_foreign
      • intermediate
      • all_free
        if first queue with a usable page is used, then the first free element from this page’s freelist is returned; if no usable page found, system adds an all_free page and retries.
        Page could be moved:
      • from intermediate to all_free
      • from all_used to intermediate
        Meta data contained **double linked list without any exploit mitigation made heap overflows very easy to exploit.
Attack

Confusing allocator to free elements from page list zones in greelist zones.
Example

Heap-Feng-Shui
Same methods

Corruption Target
All previously targeted areas still work
but double linked lists introduced by pagelist feature easiest target
(target the next/prev pointers for arbitrary writes)

iOS 8

  • Pagelits queue hardening
    • add safe unlink protection to it’s queue macros
    • double linked lists added to iOS 7 heap are now protected (how?)
    • heap overflow can no longer go after the queue pointers in metadata
  • Poisoning of larger blocks made less frequent(depending on size)
    • introduction of zp_scale feature with a default value of 4
    • poisoning of large blocks follows now the following formula
      zp_factor + element_size >> zp_scale
      this means the larger element are in a zone the less often they are poisoned example: (15/16/17) + 256 >> 4 = (31/32/33)
Attack

Heap-Feng-Shui
Same methods

Corruption Target
All previously targeted areas still work
targeting double linked lists(pagelists not possible due to safe_unlink)

iOS 9

  • vm_map_copy_t* hardening
    • its structure for kernel buffers was stripped down
      • data pointer removed because data should be after header
      • secondary size field removed because that is headerlen + size
        Personally I really don’t think this is a true harderning.
  • Repositioning of page metadata
    Shift from the end to the beginning. Possible reason: ptotect metadata against overflows
  • Randomization of initial freelist
    • when new memory is added to a zone elements within are freed one by one into the freelist
    • start with iOS9.2 the elements are no longer added linear from fist to last
    • instead, every tiem a next elements is added, a random decision is made if the first or last element should be added.
Attack

Heap-Feng-Shui
Same methods

Corruption Target
All previously targeted areas still work
only larger info leaks and arbitary pointer info leak stopped by “hardening”

iOS 10

  • more vm_map_copy_t “hardening”
  • fixed zone structure array
    • zone structure are no longer allocated on zone allocator heap(WHERE?)
    • instead stored in fixed zone_array in DATA::bss
    • so address of zone structure is now fixed, relative to kernelbase()

  • zone page metadata completely revamped

    • ALL zones make use of page metadata
    • new zone_metadata_region
    • it is reserved in the zone_map
    • keep meta data for every single page in the zone_map
    • zindex index of zone in zone_array (instead of back pointer)
    • page_count number of pages in allocation size
    • free_count number of free elements in page
    • freelist_offset offset of first free element starting from this page’s address

    • real_metadata_offset offset of this secondary meta data from page 0 metadata

      page metadata doesn’t contain pointer to head of freelist anymore, instead freelist_offset determines byte offset of first free element in page, ptr to next free block is always XORed against nopoison_cookie, backup ptr is ptr value XORed against selected cookie(nopoison/poisoned)

  • page freelist pointer leak protection
    every zone uses the new metadata allow determining correct zone by element, freeing elements into the wrong zone can be reliably detected, when this happens zfree() panics the kernel, abuse seems not possible anymore.

  • zalloc() wrappers add no inbound metadata anymore
    • new meta data can be used to determine size of elements on heap
    • heap allocation wrappers like MALLOC() and kern_os_malloc() no longer need to store the size
    • both become simpler wrappers around kalloc() and are therefore inlined in several places
    • allocations are now only application data
      (more codepaths can now be used for heap layout control)
  • zone allocator debugging features revamped
    • kernel heap allocation logging was revamped
    • no longer limited to a single zone
    • code now limited to logging in max 5 zones
    • zones selected by new kernel boot args
      zlog1, zlog2, zlog3, zlog4, zlog5
      usage: -zc zlog1=kalloc.128 zlog2=kalloc.256
    • bitfield in zone struct contains new bit 15 to select if zone_logging
    • zone struct contains zlog_btlog variable instead of global variable
    • btlog_t structure heavily modified
    • log is no longer a simple list but a hash table
Attack

Heap-Feng-Shui
Same methods, beside continued hardening of vm_map_copy_t
code paths that use MALLOC() are now also usable for heap-feng-shui(e.g. posix_spawn())

Corruption Target
application data like object and mach port pointers very interesting
BUT size field corruption to confuse free into wrong zone will trigger panic()
